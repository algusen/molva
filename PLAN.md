# PLAN.md — Pisar (Вариант 2, CLI-first: консоль → тёплый демон → Finder)

Проект **Pisar** — локальное приложение для транскрибации аудио/видео на macOS
(Apple Silicon) на базе модели `ai-sage/GigaAM-v3`. Итоговый запуск — правым кликом
по файлу в Finder; но основной интерфейс и тест-харнесс — консольная утилита `pisar`
в стиле ffmpeg.

Соглашение об именах: Python-пакет — `pisar`; CLI-команда — `pisar`;
launchd-метка — `com.<user>.pisar`; пункт в Finder — «Pisar».

Этот файл — рабочий план. Шаги передаются исполнителю по одному.
Каждый шаг имеет критерий приёмки; пока он не выполнен, к следующему не переходим.

---

## Пересмотр (важно)

CLI и демон — два фронтенда над одним ядром (`service.py`). Поэтому порядок изменён:
сначала строим CLI поверх готового ядра Phase 1, затем на нём же проверяем реальную
модель прямо из терминала, и только потом добавляем тёплый демон как оптимизацию
(CLI становится его клиентом) и триггер в Finder. Ничего из Phase 1 не выбрасывается.

## Модель исполнения

- **Контейнер (Linux)** — ядро и CLI: логика, препроцессинг ffmpeg, VAD, форматы вывода,
  разбор аргументов. Тесты — на `StubTranscriber` (без модели).
- **Mac (Apple Silicon)** — настоящая модель GigaAM-v3, launchd, Quick Action, e2e.
- Проверки: `[контейнер]` — проверяю сам; `[Mac]` — запускаете вы.

## Ключевые архитектурные решения

- Единое ядро `service.transcribe(...)`: препроцессинг → VAD → транскрайбер → сегменты.
  Все фронтенды (CLI, демон, Finder) вызывают только его.
- CLI в стиле ffmpeg: вход/выход позиционно + флаги; режим вывода в stdout для pipe.
- Два режима работы CLI:
  - **standalone** — пайплайн в процессе (просто, для тестов; холодная загрузка модели);
  - **client** — форвардит задачу работающему демону (тёплая модель, быстро).
  Пока демона нет — работает standalone; появится — client по умолчанию с фолбэком.
- Демон слушает только `127.0.0.1`; наружу ничего не уходит; передаётся путь к файлу.
- Инференс сериализуется (один процесс/модель) — лок/очередь.
- Бэкенд транскрайбера выбирается флагом/конфигом: `stub` (тесты) / `gigaam` (Mac).
- Вывод — sidecar рядом с исходником (`.txt`/`.srt`/`.vtt`/`.json`) + уведомление.
- Язык по умолчанию — русский. Точка расширения под Whisper заложена, но не реализуется.

## Технологический стек

- Python 3.11+, `venv`; FastAPI + uvicorn (демон, поздняя фаза); `click` или argparse (CLI).
- ffmpeg (16 кГц mono WAV); silero-vad (нарезка); GigaAM-v3 revision `e2e_rnnt`
  (пунктуированный нормализованный текст); transformers + torch, `trust_remote_code=True`.
- launchd LaunchAgent; Finder Quick Action как `.workflow`-бандл; terminal-notifier/osascript.

## Интерфейс CLI (стиль ffmpeg) — проектируется в Фазе 2

    pisar [ОПЦИИ] <input>...

    -f, --format txt|srt|vtt|json   формат вывода (можно повторять)
    -o, --output PATH               файл/директория (по умолчанию — sidecar рядом)
    -l, --lang ru                   язык
        --backend stub|gigaam       движок
        --revision e2e_rnnt|e2e_ctc ревизия модели
        --device auto|cpu|mps       устройство
        --vad on|off                нарезка по тишине (+ пороги)
        --stdout                    печатать текст в stdout (pipe), не писать файл
        --json                      машиночитаемый вывод (для интеграций)
        --overwrite                 перезаписывать существующие sidecar
    -v, --verbose
        --no-daemon / --daemon      режим standalone/client (появляется в Фазе 4)

    Управление демоном (Фаза 4):
    pisar daemon start|stop|status
    pisar health

# ФАЗА 2 — CLI `pisar` (стиль ffmpeg, standalone, стаб) — НОВОЕ, ТЕКУЩИЙ ФОКУС

### Шаг 2.1 — Каркас CLI и разбор аргументов
- **Цель:** утилита `pisar` с флагами, мапящимися в существующий конфиг.
- **Что делаю:** `cli.py` (click/argparse) — парсинг входов и опций из раздела выше;
  валидация аргументов; регистрация console_script `pisar` в `pyproject.toml`.
- **Артефакты:** `src/pisar/cli.py`, правки `pyproject.toml`.
- **Приёмка `[контейнер]`:** `pisar --help` показывает флаги; неверные аргументы дают
  внятную ошибку и корректный код возврата; парсинг покрыт unit-тестами.

### Шаг 2.2 — Связка CLI → service (standalone)
- **Цель:** реальный прогон пайплайна из консоли на стабе.
- **Что делаю:** маппинг флагов → объект конфига → вызов `service.transcribe(...)`;
  режимы вывода: sidecar-файлы, `--stdout`, `--json`; коды возврата (0/усп., !=0/ошибки).
- **Артефакты:** правки `cli.py`, тонкий слой над `service.py`.
- **Приёмка `[контейнер]`:** `pisar --backend stub sample.wav -f srt` пишет корректный SRT;
  `--stdout` печатает текст; `--json` даёт валидный JSON; end-to-end тест через subprocess.

### Шаг 2.3 — Батч и устойчивость
- **Цель:** поведение «как у ffmpeg» на нескольких входах.
- **Что делаю:** обработка списка файлов и glob; продолжение при ошибке одного файла
  с итоговым статусом; уважение `--overwrite`; аккуратные сообщения прогресса в `-v`.
- **Артефакты:** правки `cli.py`.
- **Приёмка `[контейнер]`:** батч из нескольких файлов (часть битые) завершается с
  корректным сводным кодом возврата; sidecar не перезатирается без `--overwrite`.

# ФАЗА 3 — Реальная модель через CLI (Mac)

### Шаг 3.1 — GigaAMTranscriber
- **Цель:** настоящая транскрибация на GigaAM-v3.
- **Что делаю:** реализация `Transcriber` (transformers, `ai-sage/GigaAM-v3`,
  revision `e2e_rnnt`, `trust_remote_code=True`); выбор устройства (CPU/MPS) с фолбэком;
  прогон VAD-сегментов и сшивка таймингов.
- **Артефакты:** `src/pisar/transcriber/gigaam.py`.
- **Приёмка `[Mac]`:** `pisar --backend gigaam --stdout clip.wav` печатает осмысленный
  пунктуированный русский текст.

### Шаг 3.2 — Выбор бэкенда и прогрев
- **Цель:** один CLI работает и со стабом, и с моделью.
- **Что делаю:** фабрика по `--backend`/env; ленивая/явная загрузка модели; понятное
  сообщение при первой (долгой) выгрузке весов.
- **Артефакты:** правки `config.py`, точки входа.
- **Приёмка `[Mac]`:** end-to-end по реальному аудио и видео (→ `.srt`) из терминала;
  длинный файл корректно режется VAD и собирается.

# ФАЗА 4 — Тёплый демон + клиентский режим CLI

### Шаг 4.1 — Демон поверх того же service
- **Цель:** держать модель в памяти.
- **Что делаю:** запуск FastAPI (`api.py`) как долгоживущего процесса; `/health`
  сообщает готовность; лок инференса; логирование.
- **Артефакты:** `scripts/run_daemon.sh`, правки `api.py` при необходимости.
- **Приёмка:** `[контейнер]` — API-тесты на стабе проходят, `/health` ок;
  `[Mac]` — демон стартует с моделью, `/health` доходит до «готов».

### Шаг 4.2 — Клиентский режим CLI
- **Цель:** `pisar` форвардит задачу демону (тёплая модель).
- **Что делаю:** авто-детект работающего демона → POST пути; фолбэк в standalone если
  демон не запущен (или `--no-daemon`); `pisar daemon start|stop|status`, `pisar health`.
- **Артефакты:** правки `cli.py`.
- **Приёмка:** `[контейнер]` — против тестового демона на стабе клиент отрабатывает;
  `[Mac]` — второй и последующие запуски заметно быстрее первого (модель тёплая).

### Шаг 4.3 — launchd LaunchAgent
- **Цель:** автозапуск/перезапуск демона.
- **Что делаю:** `.plist` (RunAtLoad, KeepAlive, env, пути логов).
- **Артефакты:** `packaging/com.<user>.pisar.plist`.
- **Приёмка:** `[контейнер]` — валидатор XML/plist ок; `[Mac]` — агент грузится,
  демон жив после перелогина, поднимается после kill.
